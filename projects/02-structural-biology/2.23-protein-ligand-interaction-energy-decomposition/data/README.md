| Field | Value |
|---|---|
| File | `sample/complex_sample.txt` |
| Origin | **Synthetic** (generated by `scripts/make_synthetic.py`; NOT real data) |
| License | Public domain (CC0) — it is synthetic |
| Size | ~3 KB |
| System | 12 protein residue beads, 4 ligand atoms, 6 trajectory frames |

This tiny file lets `demo/run_demo` run **offline, with zero downloads**, a hard
requirement for every project (CLAUDE.md §8). It is regenerated deterministically
(fixed RNG seed), so the committed sample and `demo/expected_output.txt` stay in
lockstep. To recreate or resize it:

```bash
python scripts/make_synthetic.py                          # the committed sample
python scripts/make_synthetic.py --residues 200 --frames 500   # a bigger system
```

### File format (parsed by `load_system` in `src/reference_cpu.cpp`)

Whitespace-separated text. Blank lines and lines beginning with `#` are ignored
(so the file can annotate itself). Layout:

```
F M L cutoff                         # counts + interaction cutoff (Angstrom)
<M residue rows>  name charge eps rmin_half born
<L ligand rows>        charge eps rmin_half born
<F frames>, each frame:
    <M lines>  x y z                 # residue bead coordinates (Angstrom)
    <L lines>  x y z                 # ligand atom coordinates  (Angstrom)
```

| Column | Meaning | Units |
|---|---|---|
| `F`, `M`, `L` | frame / residue / ligand-atom counts | — |
| `cutoff` | pairs farther than this contribute 0 | Angstrom (Å) |
| `name` | residue label (display only, e.g. `ARG41`) | — |
| `charge` | partial atomic charge `q` | elementary charge `e` |
| `eps` | Lennard-Jones well depth ε (per atom) | kcal/mol |
| `rmin_half` | LJ Rmin/2 (per atom, AMBER convention) | Angstrom |
| `born` | effective Born radius (how buried the atom is) | Angstrom |
| `x y z` | Cartesian coordinates | Angstrom |

### What the synthetic system encodes (the known answer)

The sample is engineered so the result is **interpretable** (PATTERNS.md §6):

- **`ARG41`** is a positively-charged arginine placed ~3.6 Å from the ligand's
  negatively-charged carboxylate oxygen — a **salt bridge**. The decomposition
  flags it as the #1 hot spot, dominated by a large favourable **electrostatic**
  component (partly offset by the Generalized-Born desolvation penalty — itself a
  teaching point).
- **`LEU88`** is a neutral, bulky leucine in close van der Waals contact — the #2
  hot spot, driven entirely by a favourable **vdW** component (zero electrostatics).
- The remaining `GLY*` residues are weak/distant background contributors.

So the per-residue table literally separates an **electrostatic** hot spot from a
**van der Waals** hot spot — the core lesson of energy decomposition.

## Full / real datasets (not redistributed here)

This is a deliberately **reduced-scope teaching version** (see `THEORY.md`
"Where this sits in the real world"). A production per-residue MM-GBSA needs a
full molecular-mechanics stack — an AMBER `prmtop`/GROMACS topology with real
force-field parameters plus an MD trajectory — which we do not ship a parser for.
`scripts/download_data.*` prints where to obtain real complexes and how to feed
them through the proper tools:

- **PDBbind** — http://www.pdbbind.org.cn — curated protein-ligand complexes with
  measured binding affinities.
- **KLIFS** — https://klifs.net — kinase-ligand structures (the oncology
  resistance-mapping use case in the catalog deep dive).
- **ChEMBL** — https://www.ebi.ac.uk/chembl/ — bioactivity data for target families.
- **ClinVar** — https://www.ncbi.nlm.nih.gov/clinvar/ — clinically observed
  resistance mutations to cross-reference against predicted hot spots.

**License:** respect every source's terms. Restricted structures must not be
committed; the offline demo always uses the synthetic stand-in above.

## Honesty / safety

The committed data is **synthetic** and labelled synthetic everywhere it appears.
Nothing here is a real protein, a real ligand, or a clinically validated binding
energy. This project is **educational only** — not for any diagnostic, treatment,
or lead-optimization decision (CLAUDE.md §1, §8).
