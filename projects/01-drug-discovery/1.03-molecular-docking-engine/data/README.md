# Data — 1.3 Molecular Docking Engine

## Committed sample (`sample/`)

| Field | Value |
|---|---|
| File | `sample/receptor_ligand_sample.txt` |
| Origin | **Synthetic** (generated by `scripts/make_synthetic.py`, no RNG) |
| License | Public domain (CC0) — it is synthetic, not derived from any real structure |
| Size | ~42 KB |
| Contents | one receptor energy grid + one rigid ligand + the pose search space |

This tiny file lets `demo/run_demo` run **offline, with zero downloads**, a hard
requirement for every project (CLAUDE.md §8). It is **synthetic** and carries **no
clinical or chemical validity** — it exists only to make the demo deterministic and
its answer verifiable by eye.

### What the sample encodes (the *known answer*)

`make_synthetic.py` builds a single Gaussian energy **well** (one attractive
pocket) at world coordinate **(0.5, −0.5, 0.0) Å**, depth 10 kcal/mol, width
σ = 1.5 Å, sampled onto a 16³ grid at 0.5 Å spacing. The ligand is a small rigid
5-atom cluster whose centroid is the origin. Because energy is summed over atoms
and the well is attractive, the best pose drops the ligand centroid into the well
— so the engine *should* recover the translation **(0.5, −0.5, 0.0)**, which it
does (see `demo/expected_output.txt`). This is the recommended way to build
synthetic data: embed a known answer and report a metric that recovers it
(`docs/PATTERNS.md` §6).

### File format

Whitespace-separated tokens; lines beginning with `#` are comments. Blocks in order:

```
GRID  nx ny nz  ox oy oz  spacing
<nx*ny*nz grid energies (kcal/mol), x fastest, then y, then z>
LIGAND  n_atoms
<n_atoms lines:  x y z weight>          # ligand-local coords (Angstrom), centroid ~ origin
SEARCH  n_trans n_rot trans_range  tcx tcy tcz
```

| Token | Meaning | Units |
|---|---|---|
| `nx ny nz` | grid points per axis | — |
| `ox oy oz` | world position of grid point (0,0,0) | Å |
| `spacing` | distance between adjacent grid points | Å |
| grid energies | energy a probe atom feels at each point | kcal/mol |
| ligand `x y z` | atom offset from the ligand centroid | Å |
| ligand `weight` | per-atom probe scale (≈ \|partial charge\|) | — |
| `n_trans` / `n_rot` | translation / rotation samples per axis | — |
| `trans_range` | ± translation swept around the pocket centre | Å |
| `tcx tcy tcz` | pocket centre (search is centred here) | Å |

Total poses scored = `n_trans³ · n_rot³` (here 7³ · 3³ = 9261).

### Regenerate or scale it

```bash
python scripts/make_synthetic.py                       # the committed tiny sample
python scripts/make_synthetic.py --n-trans 15 --n-rot 6 --n-grid 32   # bigger
```

## Full / real datasets

This teaching project uses synthetic input, but the same engine shape applies to
real docking data. The catalog points to these (study them; respect each license):

- **DUD-E** — Directory of Useful Decoys, Enhanced: 102 targets with actives +
  decoys. <https://dude.docking.org>
- **ChEMBL** — bioactivity database, >2M compounds.
  <https://www.ebi.ac.uk/chembl/>
- **PDBbind** — curated protein–ligand complexes with measured binding affinities.
  <http://www.pdbbind.org.cn>
- **CASF** — Comparative Assessment of Scoring Functions benchmark.
  <http://www.pdbbind.org.cn/casf.php>

To dock *real* structures you would (a) prepare a receptor and a ligand (e.g. with
AutoDockTools / Meeko into PDBQT), (b) precompute per-atom-type energy maps with
**AutoGrid** (the real analogue of our single synthetic grid), and (c) run a search
(AutoDock-GPU's Lamarckian GA, or Vina). `scripts/download_data.*` print these
pointers and **never** bypass any registration; `make_synthetic.py` is the offline
stand-in. See `THEORY.md` §"Where this sits in the real world".

## Honesty

Synthetic data is labeled synthetic everywhere it appears. Nothing here is a real
receptor, a real ligand, or a real binding affinity, and no output may be used for
any medical or chemical decision (CLAUDE.md §1, §8).
