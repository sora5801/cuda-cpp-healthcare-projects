| Field | Value |
|---|---|
| File | `sample/reads_haplotypes_sample.txt` |
| Origin | **Synthetic** (generated by `scripts/make_synthetic.py`, seed 5) |
| License | Public domain (CC0) — it is synthetic, not patient-derived |
| Size | < 1 KB |
| Contents | 1 truth haplotype + 2 alternative haplotypes; 8 reads drawn from the truth with ~3% simulated substitution errors |

This tiny file lets `demo/run_demo` run **offline, with zero downloads**, a hard
requirement for every project (CLAUDE.md §8). **It is SYNTHETIC: not real
sequencing data and not derived from any patient.** It exists only to exercise
the PairHMM forward math and make the GPU/CPU comparison checkable.

### File format

Lines beginning with `#` are comments and are ignored. The first non-comment
line is the header; then the haplotype rows; then the read rows.

```
# comments allowed
n_reads n_haps read_len hap_len truth delta epsilon   <- header
<hap_len-base haplotype sequence>                      <- n_haps of these
...
<read_len-base read>  q0 q1 ... q{read_len-1}          <- n_reads of these
...
```

| Header field | Meaning | Units / range |
|---|---|---|
| `n_reads` | number of sequencing reads | positive integer |
| `n_haps` | number of candidate haplotypes | positive integer |
| `read_len` | bases per read (fixed for this teaching layout) | positive integer |
| `hap_len` | bases per haplotype | positive integer (≤ 127 for the GPU build) |
| `truth` | haplotype index the reads were drawn from | `0..n_haps-1`, or `-1` if unknown |
| `delta` | pair-HMM gap-**open** probability per read base | `(0, 0.5)` |
| `epsilon` | pair-HMM gap-**extend** probability | `(0, 1)` |

Each **haplotype row** is a DNA string of `hap_len` bases (`A/C/G/T`). Each
**read row** is a DNA string of `read_len` bases followed by `read_len`
integer **Phred base-quality scores** `Q` (here a constant `Q30`, i.e. an error
probability `e = 10^(-Q/10) ≈ 1/1000`). Bases are encoded `A/C/G/T/N → 0..4` on
load; the loader and both compute paths share `encode_base()` in
`src/pairhmm_core.h`.

The synthetic sample embeds a **known answer**: every read comes from
haplotype 0, so the PairHMM likelihood `P(read | hap)` should be highest for
haplotype 0 — the demo reports "reads assigned to truth haplotype 0: 8 of 8".
Regenerate the exact committed file at any time with:

```
python scripts/make_synthetic.py          # default seed 5 -> the committed sample
```

## Full dataset (for going beyond the toy sample)

`scripts/download_data.ps1` / `.sh` print instructions and links; they do **not**
download automatically and never bypass any registration or license. The real
benchmark resources from the catalog are:

- **GiaB truth sets HG001–HG007** — gold-standard germline variant calls for
  benchmarking — https://www.nist.gov/programs-projects/genome-bottle
- **ClinVar** — clinically interpreted variants — https://www.ncbi.nlm.nih.gov/clinvar/
- **gnomAD v4** — population allele frequencies — https://gnomad.broadinstitute.org/
- **1000 Genomes high-coverage WGS** — https://www.internationalgenome.org/data

To feed real data into this teaching kernel you would extract candidate
haplotypes (from local assembly of active regions) and the overlapping reads
(from a BAM) for one locus, and convert them to the simple text format above.
That extraction is exactly what GATK's `AssemblyRegion` / `ReadThreadingAssembler`
and NVIDIA Parabricks do upstream of the PairHMM; see `THEORY.md`.

## Honesty

- The committed sample is **synthetic** and labeled synthetic everywhere.
- No output of this project is a clinical result. It is study material only
  (CLAUDE.md §1, §8). Real variant calling involves base-quality recalibration,
  local assembly, genotype-likelihood/PL computation, and joint genotyping — all
  out of scope here (described in `THEORY.md` "Where this sits in the real world").
